## 근본 원인 (Root Cause)

Toss Payments API 서버의 내부 장애로 `v1/payments/confirm` 엔드포인트가 응답 지연(timeout) 상태에 진입하였다.  
DevClass의 `TossPaymentClient`는 `@Retryable(maxAttempts=3)` 설정으로 자동 재시도하였으나 외부 장애가 지속되어 모든 재시도가 소진되었다.

---

## 영향 분석

- **결제 실패**: 결제 시도 중인 사용자에게 "결제 승인에 실패했습니다" 에러 반환
- **데이터 무결성**: `Payment` 레코드는 `PENDING` 상태로 유지 (정상, 멱등성 보장)
- **수강 등록**: 결제 미완료로 인한 수강 등록 실패 (정상, 결제 완료 후 등록)
- **이중 결제 위험**: 없음 (`orderId` unique 제약 + 멱등성 처리로 방지)

---

## 대응 조치

### 즉시 조치
1. 서비스 공지 게시 ("결제 일시 불가 안내")
2. PENDING 상태 결제 목록 추출 → 복구 후 재결제 안내 대상 확인

### 복구 후 조치
1. PENDING 상태 결제 건 사용자에게 이메일 발송 (수동)
2. Toss 측에 SLA 위반 여부 확인

## 교훈 (Lessons Learned)

1. **외부 API 장애는 불가피하다**: 재시도 정책만으로는 부족하며, Circuit Breaker + Fallback UI가 필요하다.
2. **빠른 감지가 중요하다**: Sentry 연동으로 1분 내 에러 감지가 가능했으나, Toss API 상태 페이지 자동 모니터링이 없어 원인 파악이 지연되었다.
3. **PENDING 상태 결제 관리**: 장애 종료 후 PENDING 건 처리 프로세스가 수동이었다. 자동화 배치가 필요하다.
